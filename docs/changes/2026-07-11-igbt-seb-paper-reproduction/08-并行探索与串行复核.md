# IGBT SEB 并行探索与串行复核协议修订（PA01）

状态：已生效  
修订时间：2026-07-12  
上游：[02-预注册与环境快照](02-预注册与环境快照.md)、[07-剩余工作执行计划](07-剩余工作执行计划.md)

## 1. 修订理由与主张边界

A01 核心网格减半 attempt18 完成 0–60 ns 瞬态耗时 9193 s。半步网格相对基准网格的 Collector 电流在 50 ns 和 60 ns 分别相差 4.473384% 和 15.225299%，未通过原定 2% 网格门。继续把全部锚点和诊断放在单一串行链路会造成长时间资源空闲，但这一执行效率问题不能通过回写首次预注册来消除。

因此追加本修订：

1. 保留首次预注册及其“锚点严格串行”事实，不改写首次运行预期。
2. 允许 A02–A04 先作为隔离的并行探索运行，用于评估求解刚性、耗时、输出完整性和候选物理趋势。
3. 并行探索全部闭合后，使用预先锁定的输入按 A01 → A02 → A03 → A04 严格串行确认。
4. 串行确认属于本修订下的顺序复核，不恢复“完全未偏离首次预注册”的主张。
5. 只有串行确认结果可以进入正式锚点匹配矩阵；网格风险关闭且四锚点关系通过前，扫描门保持关闭。

## 2. 证据层级

| execution_mode | 用途 | 可形成锚点结论 | 可打开扫描门 | 允许复用 |
|---|---|---:|---:|---|
| `DIAGNOSTIC_CLOSURE` | 闭合既有运行、输入审计、网格和数值诊断 | 否 | 否 | 仅复用经哈希验证的同网格检查点继续同一诊断 |
| `PARALLEL_EXPLORATORY` | A02–A04 并行摸底、资源测量和失败恢复 | 否 | 否 | 可复用冻结的同网格、同偏压成对 DC Save；不得跨 case 复用瞬态 Save |
| `SERIAL_CONFIRMATORY` | A01–A04 顺序复核 | 是 | 条件式 | 只可复用确认前已冻结且哈希一致的 DC Save；不得复用探索瞬态 Save |

每个 run 必须记录：

- `protocol_amendment_id=PA01`；
- `run_id`、`execution_mode`、`worker_id`、`case_id`、`attempt_id`、`parent_run_id`；
- `eligible_for_anchor_conclusion`；
- deck、参数、mesh、restart 主状态和 circuit 状态的 SHA-256；
- 生命周期、开始/结束时间、退出码、wall time、最后有效 checkpoint；
- 原始产物、解析 CSV 和 manifest 的路径、大小、SHA-256 与公开性；
- 失败、超时和缺失输出的明确原因，缺失值统一为 `NA`。

## 3. 固定输入与禁止项

- 并行探索和串行确认继续使用正式基准网格 SHA-256 `5c7c098706bcf07dc2656a1bfd547aea3415b02167d6afb0439c0a28f35f5265`。
- attempt18 半步网格 SHA-256 `eaffd57700f0d3b29264ff13204b51dc8d07063221cb542f7e1d20d2f650d268` 仅作为网格诊断证据，不得与正式基准网格 Save 混用。
- A02/A03 使用各自验证通过的 3000 V 成对 DC Save；A04 使用 3200 V 成对 DC Save。只有网格、物理、参数和偏压哈希完全一致时，A02/A03 才可共用只读 3000 V DC 输入。
- 每个 case 保持单线程，物理模型、HeavyIon 参数、轨迹、时间步和分类规则沿用原计划。
- 禁止修改掺杂、几何、层厚、热模型或热边界追逐论文结果；禁止把数值失败、退出或时间窗不足解释为 SEB。

## 4. 并行调度契约

- 初始最多开放两个活跃 `sdevice` 槽位；许可证席位、峰值 RSS、swap、磁盘和 I/O 通过运行前探针后才能扩容。
- 首批运行 A02 与 A04；任一槽位释放后再启动 A03。资源门不足时自动降为一个槽位，不修改物理输入。
- 每个 worker 只写唯一的本地和远端 run 目录；一个目录只允许一个进程、一套 deck 和一套文件前缀。
- 两个 worker 不得写同一 Save、stdout、manifest、CSV、Markdown 或图件。
- 出现 swap 增长、许可证拒绝、磁盘不足、checkpoint 配对缺失或相对单 case 明显负加速时，不再启动新 case；已完成 run 不重复执行。

## 5. 数据归档与单写者合并

- 私有原始数据写入 `local_runtime/igbt_seb_full_20260712_035027/runs/<run_id>/`；远端使用对应唯一目录。
- worker 只生成自己的 `run_manifest.json`、解析 CSV 和待合并行片段，不直接修改共享账本、报告或图件。
- 主线作为唯一汇总器，按 `staging → validate → merge` 顺序工作：先校验 schema、SHA、size、`NA` 和重复 run，再分配全局 `E/T/ART` ID并原子追加。
- 新增 `data/run_registry.csv` 记录运行生命周期；`case_summary.csv` 仍以 `(case_id, attempt_id)` 去重。
- 旧 `PENDING/RUNNING` 事件不得改写，只能追加 closure 事件。
- 私有原始日志、TDR、PLT、Save 和绝对远端路径不进入版本库；公开层只保存脱敏 CSV、必要图件、报告和 artifact manifest。

## 6. 执行门

1. 闭合 attempt18 并登记网格风险。
2. 通过 VM、许可证、内存、swap 和磁盘资源门。
3. 建立并冻结 3000 V 与 3200 V 成对 DC Save。
4. 完成 A02–A04 `PARALLEL_EXPLORATORY`，并由单写者合并。
5. 在查看探索结果后不改变输入；如必须改变，追加新修订并重新冻结，不能直接进入确认阶段。
6. 严格执行 A01 → A02 → A03 → A04 `SERIAL_CONFIRMATORY`，每个 case 完成归档门后才能启动下一个。
7. 只有四项确认结果定性关系均通过且网格风险获得可辩护处置，才允许进入 VCE×LET 扫描。
