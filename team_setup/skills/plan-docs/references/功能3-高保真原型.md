# 功能三 · 高保真原型 SOP

## 目的
当微观实现文档梳理完成后，据此生成**可视化、可交互**的原型，把抽象的"场景/触发/逻辑"具象成看得见、点得动的页面，固定预期、约束实现不跑偏。

## 形态（已定）
**HTML/CSS 可交互页面**（高保真）：
- 可在浏览器直接打开。
- 支持点击交互（必要时用少量原生 JS，不引入重框架）。
- 视觉接近真实产品，而非灰框线框。

## 何时执行
微观文档（功能二）就绪后，**主动询问用户**是否需要生成原型。**不要默认生成**——这是按需功能。

## 操作步骤
1. **确认范围**：问清楚要为"整个项目""某个模块"还是"某个交互点"做原型。
2. **取依据**：读取对应的微观文档，把其中的"场景/触发/逻辑/状态"作为原型的行为依据——原型必须忠实反映文档，不自行发挥新功能。
3. **生成文件**：在目标项目下建 `prototype/`（或用户指定位置），按页面/模块拆分 HTML 文件，共用一份 CSS。
4. **实现交互**：把文档里写明的"点击 X → 发生 Y"用最简方式做出来（页面跳转、显隐切换、状态变化）。
5. **可预览**：确保用浏览器打开入口文件即可走通主要交互流。
6. **回链**：在原型文件顶部注释里标注它对应哪个微观文档，保持可追溯。

## 纪律
- 原型是文档的**可视化镜像**，不是需求的二次创作。发现文档缺细节，回头补文档（功能二），不要在原型里临时编。
- 设计美观度服从于"交互逻辑清晰"——优先把流程跑通。
- 若项目已有设计规范/设计系统，遵循它。

## 关联
- **上游**：功能二的微观文档（提供行为定义）。
- **下游**：可作为功能四开发追踪、功能五任务拆分时的视觉参照。
