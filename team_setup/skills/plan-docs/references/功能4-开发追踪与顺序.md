# 功能四 · 开发追踪 + 顺序梳理 SOP

## 目的
1. **开发顺序梳理**：依据模块树 + 横向依赖，推导合理的开发先后。
2. **开发追踪**：记录每个模块的开发状态，实时更新，让人和 AI 随时知道整体进度 + 当前该做哪个。

## 开发顺序推导规则
- **被依赖者先做**：A 依赖 B，则 B 先于 A。
- **底层先于上层**：基础设施/公共模块 → 业务模块 → UI 交互。
- **纵向自底向上**：叶子能力先于聚合它们的父模块（视情况）。
- 顺序的依据来自 `03-索引.md` 的依赖总表，**不要凭空排**。若存在循环依赖，标记出来并提示用户拆解。

## 状态字段
每个模块/任务用统一状态：
- `待开发` / `进行中` / `已完成` / `受阻`（受阻需注明阻塞原因与依赖项）

## 操作步骤
1. 先定位本次需求落点：优先使用 `docs/changes/YYYY-MM-DD-需求短名/03-索引.md`；只有全局追踪才读根 `docs/03-索引.md`。
2. 读该落点 `03-索引.md` 的模块树与依赖总表。
3. 做拓扑排序，得出推荐开发顺序，写入同一落点 `04-开发追踪.md` 的"开发顺序"区。
4. 为每个模块建一行追踪条目，**直接链接到对应的微观文档**（`modules/...`）。
5. 标注初始状态（一般为 `待开发`）。
6. 随开发推进，**实时更新状态**；状态变化时同步检查是否解除了别的模块的"受阻"。

## 纪律
- 追踪表的每条都要可点进对应文档——AI 接活时顺着追踪表即可定位完整上下文。
- 顺序与依赖不一致时，以索引依赖总表为准，回头修顺序。
- 单独调用场景（"更新一下开发追踪"）：直接重算顺序 + 刷新状态即可，不必重走全程。

## 关联
- **上游**：`03-索引.md`（依赖关系）、`modules/`（被链接的文档）。
- **下游**：功能五的任务拆分按此顺序逐个推进。
