# 功能五 · 任务微观化拆分 + AI 友好提示词 SOP

## 目的
把开发任务**微观化、细节拆分，然后实现**。因为人和 AI 有"代沟"，本功能把已确认的文档翻译成**让 AI 更易理解的"源数据"提示词**，让 AI 在被约束、有细节的前提下精确执行。

## 前置条件（强制）
必须等用户**完全确认**了微观文档（功能二）+ 开发追踪文档（功能四）之后，才进入此步。未确认不得开始拆任务。

## 任务粒度
拆到**极细**——哪怕"加一条 debug 语句""改一个字段名"也算一个独立任务。每个任务对应一个明确、可单独完成的动作。

## 提示词本质
- 是**给 AI 的源数据**，不是给人看的文案。**贴合 AI 理解习惯，不堆砌辞藻、不追求复杂。**
- 必须**锚定到具体文档**：引用对应的微观文档/模块路径，让 AI 有精确依据，不发散。
- 包含：要做什么、改哪个文件/模块、依据哪个文档、完成的判定标准。

## 两种交付方式
### 方式 A —— 直接复制发送（基础形态，必须支持）
生成一段可直接复制、粘贴发给 AI 的提示词，用户手动驱动逐个任务完成。

### 方式 B —— `/goal` 自动执行（可选增强，非必须）
把提示词写成带**可验证完成条件**的形式，交给 `/goal` 自主逐个完成。
- `/goal`（Claude Code 与 Codex CLI 均内置）= 目标驱动自主执行：给一个可验证完成条件，AI 多轮自主执行，独立检查模型逐轮校验，达成即停。
- 因此方式 B 的提示词**必须**含可被机器/检查模型判定的完成条件（如"X 文件新增 Y 函数，且 Z 测试通过"）。

## 操作步骤
1. 确认前置条件已满足（用户已确认文档）。
2. 按 `04-开发追踪.md` 的开发顺序，取下一个待开发模块。
3. 把该模块的微观文档拆成一串极细任务。
4. 为每个任务生成提示词（默认方式 A；用户要自动化则同时给方式 B 的可验证完成条件版）。
5. 提示词里务必带上：关联文档路径、改动范围、完成判定标准。
6. 任务完成后，回到功能四更新开发追踪状态。

## 提示词模板（方式 A）
```
任务：<一句话目标>
依据文档：docs/modules/.../xxx.md
改动范围：<文件/模块>
具体要求：
- <步骤1>
- <步骤2>
完成标准：<可判定的结果>
```

## 提示词模板（方式 B · /goal）
```
目标：<一句话目标>（依据 docs/modules/.../xxx.md）
完成条件（可验证）：<例如：新增 foo() 且 npm test 通过、页面点击 X 后出现 Y>
约束：仅改动 <范围>，不改其它模块。
```

## 关联
- **上游**：功能二微观文档、功能四开发追踪与顺序。
- **下游**：执行结果回写功能四状态，形成闭环。
