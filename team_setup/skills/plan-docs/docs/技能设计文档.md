# 技能设计文档：项目规划与文档化技能（plan-docs）

> 本文档把与用户讨论确定的五大功能、文档体系规范、工作流程与触发逻辑固化下来，作为后续实现 skill 的蓝图。
> 一切设计依据来自 [用户原话.md](./用户原话.md)，如有冲突以用户原话为准。

---

## 一、技能要解决的核心问题

**核心目的有两个，互为表里：**

1. **约束 AI** —— 主动给 AI 套上明确的边界与依据，让它只在被界定的范围内行动，不自由发挥。
2. **把功能开发细节化** —— 把需求与实现拆解到足够细的颗粒度（细到"点这个按钮的逻辑"、细到"一条 debug 语句"），让每一步都有清晰、可执行的定义。

大多数人只跟 AI 沟通**大方向**、不梳理细节，AI 便缺乏约束、缺乏细节依据，从而自由发挥、产出跑偏（上下文超限导致的发散/偏移只是其中一种表现，而非核心问题本身）。

本技能的本质是：**为项目建立一套从宏观到微观、有索引、前后逻辑通顺且互相关联的文档网络，用它来约束 AI 并承载被细节化的功能开发**，并以此驱动需求讨论、原型设计、开发追踪与任务执行——让 AI 在任意时刻都有明确约束与精确定义可循，而非依赖自由发挥或会被截断的记忆。

设计上贯穿一条原则：**人和 AI 有"代沟"**——人用原始口语表达，AI 理解有偏差。因此技能要不断把"人话"翻译/收敛成"AI 能精确执行、可自检"的结构化文档与提示词，这正是"约束 + 细节化"的落地手段。

## 二、五大功能

### 功能一：保留用户原话
- **做什么**：在协作全程，逐字、忠实地记录用户说的每一句话（不改写、不总结），写入 `docs/用户原话.md`。
- **为什么**：需求可溯源；当后续梳理/文档与用户本意出现偏差时，能回到原始表述对齐。
- **要点**：按时间/日期分段追加；AI 自己的查证结论等非用户原话，需明确标注区分。

### 功能二：PM 视角需求讨论 + 分层文档体系
- **做什么**：站在**产品经理**角度，和用户讨论需求与**产品实用性**（而非一上来就拆开发任务），并把讨论结果沉淀为**分层文档体系**。
- **关键纪律**：细到**每个功能模块**、甚至**"点这个按钮是什么场景、什么逻辑"**，哪怕只有几十个字，**也单独成文档**；但绝不是散落的碎文件，而是**有索引、可前后追溯、互相关联**的体系。
- **关联关系（C 方案：中心索引 + 文档内上下游）**：
  - **纵向（包含/父子）**：功能模块 → 子模块 → 子子模块……**可无限递归**，这棵树本身就是"开发关联关系"。
  - **横向（依赖/调用）**：跨树枝的触发/调用关系（如"按钮X"触发"模块B的某流程"）。
  - **索引（单一事实来源）**：`03-索引.md` 用缩进画出全局模块树 + 横向依赖总表，改结构以它为准。
  - **每个文档头部统一标注**：`上级` / `下级` / `依赖`，逐层可追。

### 功能三：按需生成高保真原型
- **做什么**：当微观实现文档梳理完成后，**主动询问**用户是否需要据此生成原型图。
- **形态**：**HTML/CSS 可交互页面**（高保真，可在浏览器打开并点击交互）。
- **为什么靠谱**：微观文档已写清"场景/触发/逻辑"，原型有精确依据，不会乱画。

### 功能四：开发追踪 + 开发顺序梳理
- **做什么**：
  1. **开发顺序梳理**：依据功能二的模块树 + 横向依赖，推导合理的开发先后（被依赖者先做、底层先于上层）。
  2. **开发追踪文档**：记录每个模块的状态（待开发/进行中/已完成/受阻），实时更新。
- **关联**：开发顺序从 `03-索引.md` 的依赖关系推导；追踪表每条**直接链接到对应微观文档**，AI 接活时顺着追踪表即可定位完整上下文。

### 功能五：任务微观化拆分 + AI 友好提示词生成（执行层）
- **前置条件**：用户**完全确认**微观文档 + 开发追踪文档之后，才进入此步。
- **核心做什么**：把开发任务**微观化、细节拆分，然后实现**——AI 把任务**拆到极细**（哪怕"加一条 debug 语句"也算一个独立任务），为每个任务生成一段提示词。
- **提示词的本质**：是**让 AI 更易理解的"源数据"**——贴合 AI 的理解习惯，不追求复杂华丽，目的就是把"人话"翻译成 AI 能精确执行的输入。
- **交付/使用方式（`/goal` 非必须，二选一或并存）**：
  - **方式 A — 直接复制发送**：用户把生成的提示词直接复制，粘贴发给 AI 去完成某个小任务。**这是基础形态，必须支持。**
  - **方式 B — `/goal` 自动执行**：把提示词写成带**可验证完成条件**的形式，交给 `/goal` 自主逐个完成。**这是可选增强，不强制。**
- **关于 `/goal`（已查证，作为可选方式 B 的依据）**：Claude Code 与 OpenAI Codex CLI 均内置 `/goal`——目标驱动的自主执行模式：给一个**可验证的完成条件**，AI 自主多轮执行，独立检查模型逐轮校验，达成即停（Claude Code 默认用 Haiku 校验）。故方式 B 的提示词需带明确可验证完成条件以契合该机制。

## 二.五、五大功能 ↔ 两大核心目的

每个功能都同时服务于「约束 AI」与「细节化功能开发」两个核心目的：

| 功能 | 如何"约束 AI" | 如何"细节化功能开发" |
|------|--------------|---------------------|
| 一 原话留存 | 提供可溯源的本意基准，约束 AI 不偏离用户真实意图 | — |
| 二 分层文档+递归模块树 | 划定边界、明确每个模块职责与依赖 | 拆到功能模块/子模块/"按钮逻辑"级别 |
| 三 高保真原型 | 用可视化交互固定预期，约束实现不跑偏 | 把交互细节具象成可点击的页面 |
| 四 开发追踪+顺序 | 约束开发范围与节奏（该做哪个、做到哪） | 按模块逐条追踪状态 |
| 五 任务微观化+AI友好提示词 | 用"源数据"提示词（+可选 `/goal` 校验）约束 AI 只做该做的 | 拆到"一条 debug 语句"级别的任务 |

## 三、文档体系规范

```
docs/
├── 用户原话.md            ← 功能一：原始输入，溯源用
├── 00-项目说明书.md       ← 这是什么项目、为谁做、解决什么、目标与边界
├── 01-需求文档.md         ← 产品需求总览（功能清单、优先级、用户场景）
├── 02-架构文档.md         ← 技术架构、技术栈、模块划分、数据流
├── 03-索引.md             ← 全局导航：模块树（缩进展示父子）+ 横向依赖总表【单一事实来源】
├── 04-开发追踪.md         ← 功能四：开发顺序 + 状态看板（链接到各模块文档）
├── 05-术语表.md           ← 统一项目内术语/命名，避免人与 AI 叫法不一
├── 06-数据字典.md         ← 字段/数据结构定义
├── 07-API文档.md          ← 接口清单与出入参定义
├── 08-测试用例.md         ← 各模块/交互点验收用例（也可作 /goal 的可验证完成条件来源）
└── modules/               ← 功能二：微观实现文档（可无限递归）
    └── 模块A/
        ├── _A.md          ← 模块说明（含上级/下级/依赖）
        ├── 子模块A1/
        │   ├── _A1.md
        │   ├── 子模块A1a/
        │   │   └── _A1a.md
        │   └── 按钮X.md   ← 最小叶子文档（场景/触发/逻辑/上级/依赖）
        └── 子模块A2/...
```

**层与层的引用链（"前后逻辑通顺"的落地）：**
- 项目说明书 → 决定 → 需求文档有哪些功能
- 需求文档的每个功能 → 被分配到 → 架构文档的某个模块
- 架构文档的模块划分 → 对应 → `modules/` 目录树
- 索引 → 把以上全部串成可跳转的导航网
- 开发追踪 + `/goal` 任务 → 锚定到 → 具体微观文档

**微观文档头部模板（示例）：**
```
上级：[[_A1]]
下级：[[子模块A1a]]、[[按钮X]]
依赖：[[模块B-流程Y]]   ← 横向，可空
---
场景：……
触发：……
逻辑：……
```

## 四、触发与调用方式

**两种入口模式：**
- **新项目 → 引导式**：启动后按顺序走全程
  `原话留存 → PM需求讨论&分层文档 → 高保真原型 → 开发顺序&追踪 → 任务微观化&AI友好提示词`
- **老项目 → 初始化式**：AI 先主动了解现有项目（读代码/结构），**逆向构建**出本文档树，让用户借此更了解项目，再去完善与继续开发。

**调用粒度（两者都支持）：**
- **引导式串行**：新项目从头到尾走一遍完整流程。
- **单独调用**：任意功能可独立触发，无需重走全程，例如"更新开发追踪""新增一个模块文档""只生成某模块的原型""为某任务生成提示词"。

## 五、整体工作流（闭环）

```
[原话留存]
   ↓ 持续贯穿全程
[PM需求讨论] → 产出 00/01/02/03 + modules/ 微观文档
   ↓ 询问
[高保真原型(HTML/CSS)] —— 可选
   ↓
[开发顺序梳理 + 开发追踪(04)]
   ↓ 用户完全确认文档后
[任务微观化拆分 → AI友好提示词(源数据)] → A.直接复制发给AI / B.(可选)交给/goal自动跑
```

## 六、已定实现决策

- **调用方式**：引导式串行 + 单独调用，两者都支持。
- **文档体系**：在 6 类核心文档（原话/说明书/需求/架构/索引/开发追踪 + 模块树）基础上，默认含**术语表、数据字典、API 文档、测试用例**4 类扩展文档。
- **skill 落地形态**：`SKILL.md`（主调度）+ **多个子文档**（各功能 SOP、文档模板等）。命名暂定 `plan-docs`。
